# Design Notes

这份文档记录线程池当前版本的几个设计取舍。源码里只保留并发不变量和容易写错的边界，这里放更适合展开说明的背景。

## 任务结果：从 Any/Result 到 std::future

早期实验版曾考虑过自定义 `Any` / `Result` 一类的结果包装，用来保存任意返回值和异常。这个方向能练习类型擦除，但线程池本身会很快被“结果系统”淹没：

- 需要自己处理返回值生命周期；
- 需要额外设计异常传播；
- 调用方还要学习一套项目自定义 API；
- 实现越复杂，越难判断问题来自线程池还是结果包装。

当前版本改成 `std::future` + `std::packaged_task`。这样任务返回值、`void` 返回值和异常传播都交给标准库处理，线程池只负责调度和生命周期。这也让 `submit()` 的语义更接近标准库用户的直觉。

## 任务队列为什么存 std::function<void()>

用户提交的任务可能返回 `int`、`std::string`、`void` 或其他类型，但工作线程只需要“取出一个任务并执行它”。因此队列里统一保存 `std::function<void()>`。

具体返回值由外层的 `std::packaged_task<ReturnType()>` 写入对应的 `std::future<ReturnType>`。这样队列不需要知道任务的真实返回类型。

## submit() 为什么拷贝或移动参数

`submit()` 返回后，真正执行任务的是工作线程。如果只保存调用方栈上的引用，很容易出现悬空引用。

所以当前实现会把函数对象和参数移动或拷贝进任务闭包。需要引用语义时，调用方必须显式使用 `std::ref` 或 `std::cref`。这比隐式保存引用更啰嗦一点，但边界更清楚。

## cached 线程退出后为什么还要回收 std::thread

cached 模式里，多出来的线程空闲超时后会退出。退出时只减少 `currentThreadSize_` 还不够，因为 `threads_` 容器里仍然留着对应的 `std::thread` 对象。

一个已经结束但仍然 `joinable()` 的 `std::thread` 不能直接销毁，否则程序会调用 `std::terminate()`。因此 worker 退出时会标记 `WorkerState::finished`，线程池随后在提交或扩容路径中调用 `reapFinishedThreadsLocked()`，把这些线程 `join` 并从容器中移除。

这里有一个重要细节：`join` 可能阻塞，不能在持有 `taskQueMtx_` 时执行，否则 worker 退出或 `submit()` 路径可能被同一把锁卡住。

## shutdown 为什么不直接清空队列

`shutdown()` 的语义是“停止接收新任务，但尽量完成已经提交的任务”。如果直接清空队列，调用方手里的 `future` 可能永远等不到结果，或者需要额外设计一套取消/失败通知机制。

当前版本选择更朴素的语义：

1. `isPoolRunning_` 置为 `false`，拒绝新任务；
2. 唤醒等待中的 worker 和提交线程；
3. worker 继续消费已经入队的任务；
4. 队列清空后 worker 退出；
5. `shutdown()` join 所有 worker。

缺点也很明确：如果已经入队的任务永久阻塞，`shutdown()` 也会一直等待。

`shutdown()` 被定义为外部生命周期操作。任务内部不允许调用它，因为 worker 等待自身退出、对象析构和资源回收这几个动作会纠缠在一起，容易留下悬空访问或死锁边界。

同理，`ThreadPool` 对象本身也不应该在线程池自己的 worker 任务内部销毁。析构函数会尽力走不抛异常的关闭流程，但从自身 worker 中销毁线程池没有可恢复的安全语义。

## shutdown 后为什么允许再次 start

`shutdown()` 完成后，线程对象已经 join 并从容器中移除，线程计数也会归零。因此同一个 `ThreadPool` 对象可以再次调用 `start()` 创建新的一组 worker。

这个语义让 `shutdown()` 更像一次完整的“停止当前运行周期”，而不是销毁线程池对象。配置接口仍然要求在每次 `start()` 前调用；关闭完成后对象不处于运行状态，所以可以重新配置后再启动。

## submit 超时为什么默认 1 秒

队列满时，提交线程最多等待配置的超时时间，默认值是 1 秒。这个默认值不是性能最优解，而是为了让拒绝策略简单、可观察，也方便写测试。

如果直接无限等待，调用方很难区分“系统只是慢”和“任务已经无法提交”。如果完全不等待，在短暂突发流量下又容易过早拒绝任务。1 秒是这个学习项目里比较折中的默认行为。

如果调用方希望更快失败，或者愿意为短暂突发等待更久，可以在 `start()` 前调用 `setSubmitTimeout(std::chrono::milliseconds)` 调整这个值。

## 当前没有做的事情

当前版本没有实现：

- 任务取消；
- 任务优先级；
- work stealing；
- 暂停/恢复线程池；
- 每个任务独立设置超时时间；

这些功能都可以继续扩展，但每一个都会改变并发语义。现在先把“提交、执行、返回结果、拒绝、关闭、cached 回收”这条主线做清楚。
